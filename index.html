AABB包围盒和OBB包围盒

AABB 包围盒碰撞检测

适用于快速初步碰撞检测（Broad Phase）。

二维空间中，AABB 是一个矩形；三维空间中，AABB 是一个立方体。

不支持旋转，因此对复杂形状或旋转物体的检测精度较低。

碰撞检测条件：

两个 AABB 相交的条件是：在所有轴上，它们的边界都有重叠。

具体条件如下：

  • 在 X 轴上：A.maxX ≥ B.minX 且 A.minX ≤ B.maxX
  • 在 Y 轴上：A.maxY ≥ B.minY 且 A.minY ≤ B.maxY
  • 在 Z 轴上（三维空间）：A.maxZ ≥ B.minZ 且 A.minZ ≤ B.maxZ

OBB 包围盒碰撞检测

分离轴定理（SAT）：通过投影到多个分离轴上，判断两个 OBB 是否相交

具体步骤：

  1. 计算两个 OBB 的中心点和轴向。
  2. 将每个 OBB 的顶点投影到这些轴上。
  3. 检查投影区间是否有重叠。如果所有轴上的投影都有重叠，则两个 OBB 相交

判断点是否在三角形内

AP,BP,CP分别叉乘,之后点乘判断方向,即点乘为正值,点p在三角形内部

求解一个向量在另一个向量的投影

a · b = ||a|| ||b|| cosθ 点乘公式

首先计算向量夹角 θ = arccos[(a·b)/(||a|| ||b||)]

然后可得 proj = b cosθ

简化公式可得

如果 b 是单位向量，计算可简化为：proj = (a · b) * b

已知入射向量和法线向量求解反射向量

向量加减法,向量投影法

ON = 2S (S为法向量的一半)

R = I + 2S (画图可知,向量加法)

因为S 是 -I在N的投影

S = - (I · N)N

所有 R=I−2(I⋅N)N

求解一个向量在一个平面上投影

计算向量 v 在法向量 n 上的投影向量 vproj-n

从向量 v 中减去 vproj-n，得到向量 v 在平面上的投影向量 vproj

如果法向量 n 是单位向量，即其长度为 1，则 n⋅n=1，公式简化为：

v_proj = v - (n·v)n

判断一个点是否在平面之上

点法式平面方程：n·(P - P₀) = 0

  • n 是平面法向量（需归一化）
  • P₀ 是平面上已知点
  • P 是待测试点

判断一个点在是否在视锥之内

n·(P - P₀) >= 0 则说明点和法向量方向相同

然后依次判断六个面,符合条件则说明在视锥之内

Upvalue和Closure

基本概念

Closure

在 Lua 中，闭包是一个函数加上它可以访问的非局部变量（upvalue）的组合。闭包允许函数记住并访问它被创建时的环境。

Upvalue

Upvalue 是指被嵌套函数访问的外部函数的局部变量。这些变量在外部函数返回后仍然可以被内部函数访问。

工作原理

在这个例子中：

  • outer 函数创建了一个局部变量 x
  • 返回的匿名函数捕获了这个 x 作为 upvalue
  • 即使 outer 已经返回，闭包仍然可以访问和修改 x

关键特性

Upvalue 的生命周期

  • Upvalue 的生命周期与捕获它的闭包绑定
  • 当最后一个引用该 upvalue 的闭包被回收时，upvalue 才会被回收

与普通局部变量的区别

特性 普通局部变量 Upvalue
生命周期 函数执行期间 至少与闭包生命周期相同
访问速度 较快（栈访问） 稍慢（需要间接访问）
共享性 不共享 可被多个闭包共享
存储位置 栈 堆（闭包创建后）

Lua 表的 Rehash 机制

在 Lua 中，表（table）是动态数据结构，当表的大小发生变化时，Lua 会自动执行 rehash（重新哈希）操作来优化表的存储和访问效率。

触发 Rehash 的条件

Lua 表由 数组部分（array part） 和 哈希部分（hash part） 组成，rehash 主要发生在以下情况：

插入新元素时

  • 当哈希部分的负载因子（元素数量/槽位数量）超过阈值（通常为 0.5~0.75）
  • 例如：向表中插入大量新键值对时

删除大量元素时

  • 当哈希部分的元素数量远少于槽位数量（内存浪费）

表的数组部分和哈希部分需要重新平衡

  • Lua 会根据元素类型（连续整数键 → 数组部分；其他键 → 哈希部分）动态调整

总结

  • Rehash 是 Lua 表动态调整存储结构的核心机制
  • 预分配大小和键类型规划可显著减少 rehash 开销
  • 在性能敏感场景（如游戏主循环）中，应避免密集触发 rehash

Table的构成

  1. 数组部分（Array Part）
    • 用于存储 连续整数索引（1, 2, 3, …） 的值。
    • 类似于传统数组，访问时间复杂度为 O(1)。
    • 适用于顺序存储的场景（如 t[1], t[2], …）。
  2. 哈希部分（Hash Part）
    • 用于存储 非连续整数索引、字符串键或其他类型的键。
    • 使用 开放寻址（Open Addressing） 的哈希表实现。
    • 平均时间复杂度为 O(1)（最坏情况下可能退化到 O(n)）。

存储优化策略

Lua Table 会根据键的类型和分布 动态调整存储方式：

  • 如果键是连续整数（1, 2, 3, …），优先存储在 数组部分（访问更快）。
  • 如果键是非连续整数、字符串或其他类型，存储在 哈希部分。
  • 如果数组部分有空位，Lua 可能会将部分哈希数据迁移到数组部分以提高性能。
